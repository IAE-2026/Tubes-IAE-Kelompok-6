# Justifikasi Transaksi Kritis Services C
Jadi pada services saya ini yaitu membership & voucher terdapat get menampilkan data member, post membuat member baru dan get menampilkan data member berdasarkan code member, maka dari itu sangat penting menambahkan keamanan dalam siapa saja yang boleh membuat member baru karena, dalam membuat member baru terdapat field "membership_type" yang menentukan berapa jumlah diskon yang didapatkan setiap member, hal tersebut dapat mempengaruhi data di dalam database, jika hal ini tidak di validasi dan diperiksa dengan benar akan mengakibatkan manipulasi data yang menyebabkan kerugian pada sistem parkir/ dapat merugikan sistem parkir (perusahaan) itu sendiri.

Maka dari itu diperlukan validasi role/user dengan menggunakan token yang JWT SSO, yang dicatat di sistem SOAP, lalu akan dikirimkan statusnya melalui RabbitMQ.

# Alur sistem nya (diagram sequence)
admin -> membawa informasi data (termasuk penentuan membership_type) & Token JWT SSO -> POST /api/v1/memberships (membuat memberships) -> Service C -> akan memverifikasi token JWT menggunakan kunci publik dari SSO Dosen (GET /api/v1/auth/jwks) dan mengecek role pengguna -> Service C -> Menyimpan data member baru tersebut ke dalam -> Database MySQL Lokal -> Service C -> Membentuk dan mengirimkan catatan audit (format XML) -> SOAP Dosen (/soap/v1/audit) -> memberikan respons dengan bukti pencatatan -> Service C -> membentuk status notifikasi member baru dengan menggunakan format json -> Mengirimkan Status Pembuatan Member -> RabbitMQ (iae.central.exchange) -> service C -> mengembalikan informasi berhasil beserta datanya ke -> admin
